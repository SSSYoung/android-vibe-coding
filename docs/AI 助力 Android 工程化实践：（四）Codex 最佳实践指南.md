# 最佳实践

开始使用 Codex，并掌握那些已经被验证能带来更好结果的实践方法。

无论你是在 [CLI](https://developers.openai.com/codex/cli)、[IDE 扩展](https://developers.openai.com/codex/ide) 还是 [Codex 应用](https://developers.openai.com/codex/app) 中使用 Codex，本页都整理了一套能持续提升产出质量的核心习惯。它们覆盖了提示词、规划、`AGENTS.md`、配置、MCP、skills、自动化，以及如何把 Codex 纳入你的日常开发流程。

最有效的用法，是把 Codex 当成一个可以持续配置和优化的团队成员，而不是一次性的助手。一个实用的思路是：为任务提供恰当的上下文，用 `AGENTS.md` 固化关键指引，让配置贴合你的工作方式，用 MCP 连接外部系统，把重复流程沉淀为 skills，并把稳定流程交给自动化运行。

## 从提示词和上下文开始

即使提示词并不完美，Codex 往往也能产出有用的结果。很多时候，你只要把问题抛出来，就已经能得到不错的输出。但在更大的代码库里，或者面对风险更高的任务时，清晰的[提示词](https://developers.openai.com/codex/prompting)会显著提升结果的可靠性。

一个稳妥的默认做法，是在提示词里明确四件事：

- **目标**：你希望改什么，或者构建什么？
- **上下文**：哪些文件、目录、文档、示例或报错与这项任务相关？你也可以用 @ 引用文件来补充上下文。
- **约束**：Codex 需要遵守哪些标准、架构、安全要求或团队约定？
- **完成标准**：什么条件满足后，这项任务才算完成？比如测试通过、行为符合预期，或某个 bug 不再出现。

这样可以让 Codex 更聚焦，减少无谓猜测，也更方便你后续审查结果。

根据任务复杂度选择合适的推理级别，并观察哪种设置最适合你的工作流：

- **Low**：适合快速、范围明确的任务
- **Medium** 或 **High**：适合更复杂的修改或调试
- **Extra High**：适合耗时较长、代理式、推理密集的任务

在 Codex 应用里，你也可以直接用语音听写，把想让 Codex 做的事说出来，而不是逐字输入。

## 复杂任务先规划

如果任务本身复杂、模糊，或者一时难以说清楚，先让 Codex 规划，再进入编码。

对大多数人来说，最简单有效的方式就是用 Plan 模式。它会让 Codex 先收集上下文、提出澄清问题，并在真正动手前形成更扎实的计划。你可以用 `/plan` 或 <kbd>Shift</kbd>+<kbd>Tab</kbd> 切换到这个模式。

如果你脑子里只有一个大概方向，还说不清怎么描述，也可以直接让 Codex 来“采访”你。先让它挑战你的假设，把模糊的想法一步步整理成可以执行的方案，再开始写代码。

如果你的工作流更复杂，还可以让 Codex 在长时任务或多步骤任务里遵循 `PLANS.md` 或执行计划模板。详细说明可以参考[执行计划指南](https://developers.openai.com/cookbook/articles/codex_exec_plans)。

## 用 `AGENTS.md` 复用指引

当你发现某种提示方式反复有效，下一步就不该继续手动重复，而是把它写进 `AGENTS.md`。Codex 会自动读取它，把里面的指引纳入上下文。

可以把 `AGENTS.md` 理解成一份面向代理的开放式 README。它最适合用来说明：在这个仓库里，Codex 应该如何工作。

一份好的 `AGENTS.md` 通常会包含：

- 仓库结构和关键目录
- 如何运行项目
- 构建、测试和 lint 命令
- 工程约定和 PR 预期
- 约束条件和禁止事项
- 完成标准，以及如何验证工作结果

你可以用 `/init` slash 命令，在当前目录里快速生成一个起步版的 `AGENTS.md`。这会是一个不错的起点，但你仍然应该按团队的真实构建、测试、审查和发布方式对它做进一步调整。

`AGENTS.md` 也可以按层级拆分：

- 全局 `AGENTS.md`（位于 `~/.codex`），用于个人默认设置
- 仓库级文件，用于团队共享规范
- 子目录下更具体的文件，用于局部规则

如果某个更具体的文件离你当前所在目录更近，那么那份指引会优先生效。

保持它简洁实用。相比一份又长又泛的规则清单，一份短小、准确的 `AGENTS.md` 通常更有价值。先写最基本的内容，只有当你反复遇到同类问题时，再补充新的规则。

如果 `AGENTS.md` 逐渐变大，就让主文件保持精简，并把任务相关的细节拆到单独的 Markdown 文件里，例如规划文档、代码审查规范或架构说明。

如果 Codex 连续两次犯同样的错误，就让它回顾原因，并据此更新 `AGENTS.md`。这样写进去的规则才会真正来源于实际摩擦，而不是抽象设想。

## 配置 Codex，获得更一致的结果

配置是让 Codex 在不同会话和不同平台上保持稳定表现的关键。比如，你可以预先设定模型选择、推理投入、sandbox 模式，以及审批策略的默认值。

一个常见的起步方式是：

- 把个人默认设置放在 `~/.codex/config.toml` 中（Settings → Configuration → Open config.toml from the Codex app）
- 把仓库特有的行为配置写到 `.codex/config.toml`
- 只在临时场景下用命令行参数覆盖配置（如果你使用 CLI）

[`config.toml`](https://developers.openai.com/codex/config-basic) 是定义持久化偏好的地方，比如 MCP 服务器、profile、多代理设置，以及功能开关。你既可以手动编辑，也可以直接让 Codex 帮你修改。

Codex 自带操作系统层面的 sandboxing，其中有两个关键开关值得你关注。Approval 模式决定 Codex 在什么情况下需要向你请求命令执行许可；sandbox 模式则决定 Codex 能否读取或写入目录，以及它可以访问哪些文件。

如果你刚开始接触编程代理，建议先使用默认权限。先保持审批和 sandboxing 比较严格，只有在明确需要时，再为可信仓库或特定工作流适度放宽。

> 尽早把 Codex 配置到贴近真实工作环境的状态。很多看似是“质量问题”的现象，实际上是 setup 问题，例如工作目录不对、缺少写权限、模型默认值不合适，或者缺少必要的工具和连接器。

## 用测试和审查提升可靠性

不要在 Codex 完成修改后就停下来。你应该让它在需要时创建或更新测试、运行相关检查、确认结果是否正确，并在接受成果前进行审查。

Codex 可以帮你跑完这个闭环，但前提是它知道什么叫“做得对”。这些标准可以直接写在提示词里，也可以写进 `AGENTS.md`。

这些检查通常包括：

- 为变更补充或更新测试
- 运行正确的测试套件
- 检查 lint、格式化或类型检查结果
- 确认最终行为和你的要求一致
- 审查 diff 中是否存在 bug、回归或风险模式

在 Codex 应用中，你可以切换到 diff 面板，直接在本地[审查变更](https://developers.openai.com/codex/app/review)。点击具体代码行留下反馈后，这些反馈会自动进入下一轮 Codex 对话的上下文。

另一个常用入口是 `/review` slash 命令，它支持多种审查方式：

- 基于 base 分支做 PR 风格审查
- 审查尚未提交的改动
- 审查单个 commit
- 使用自定义审查指令

如果你们团队已经有 `code_review.md`，并且在 `AGENTS.md` 中引用了它，那么 Codex 在审查时也可以遵循这套规范。对于希望在不同仓库、不同贡献者之间保持审查风格一致的团队来说，这种做法很有效。

> Codex 不应该只负责生成代码。只要给出合适的指令，它同样可以参与 **测试、检查和代码审查**。

## 用 MCP 获取外部上下文

当 Codex 需要的信息不在仓库内部时，就该用 MCP。它能把 Codex 接到你已经在使用的工具和系统上，这样你就不必一次次把实时信息手动复制到提示词里。

[Model Context Protocol](https://developers.openai.com/codex/mcp)，也就是 MCP，是一个把 Codex 与外部工具和系统连接起来的开放标准。

Codex 支持 STDIO 和 Streamable HTTP 服务器，也支持 OAuth。

在 Codex 应用中，你可以进入 Settings → MCP servers 查看自定义服务器和推荐服务器。很多情况下，Codex 还可以直接帮你完成所需服务器的安装，你只需要开口请求即可。如果你使用 CLI，也可以通过 `codex mcp add` 命令添加自定义服务器，包括名称、URL 和其他配置项。

只在工具确实能解锁真实工作流时再接入它们。不要一开始就把所有在用的工具全部连进来。先从一两个能明确减少手工重复操作的工具开始，再逐步扩展。

## 把重复工作沉淀为 skills

当某个工作流已经足够稳定、会重复出现时，就不要继续依赖冗长的提示词或反复来回沟通了。你可以用 [Skill](https://developers.openai.com/codex/skills) 把指令、上下文和辅助逻辑封装进 `SKILL.md`，这样 Codex 就能更稳定地重复使用它们。skills 可以在 CLI、IDE 扩展和 Codex 应用中使用。

让每个 skill 只专注一项工作。先从两三个典型用例开始，定义清楚输入和输出，再写好描述，让人一眼就知道这个 skill 是做什么的、什么时候该用它。

不要一开始就试图覆盖所有边缘情况。先选一个有代表性的任务，把流程打磨到足够可靠，再把它抽象成一个 skill，并在后续使用中持续改进。只有在脚本或附加资源确实能提升可靠性时，才把它们加入 skill。

一个简单的经验法则是：如果你反复复用同一段提示词，或者反复纠正同一类流程，那它就值得被做成一个 skill。

下面这些重复性工作尤其适合做成 skills：

- 日志分类
- 发布说明草稿
- PR 审查清单
- 迁移规划
- 遥测或事件摘要
- 标准化调试流程

`$skill-creator` skill 是创建第一个版本 skill 的好起点。先在本地迭代；等打磨成熟、准备共享时，再把它封装成 [plugin](https://developers.openai.com/codex/plugins/build)。一个 skill 最关键的部分之一，就是它的描述字段。这个描述需要明确说明：它做什么，以及什么时候使用。

个人 skills 存放在 `$HOME/.agents/skills` 中；团队共享的 skills 可以放在仓库内的 `.agents/skills`。这对新成员入职尤其有帮助。

## 用自动化处理重复流程

一旦某个工作流稳定下来，你就可以让 Codex 在后台按计划运行它。在 Codex 应用中，[automations](https://developers.openai.com/codex/app/automations) 可以让你选择项目、提示词、执行频率和运行环境，把某项重复任务交给系统定期执行。

当一项任务开始频繁重复时，你可以在 Automations 标签页中创建自动化。你可以指定它在哪个项目里运行、使用什么提示词（也可以调用 skills），以及运行频率。你还可以决定它是在专用的 git worktree 中运行，还是直接在你的本地环境里运行。关于这部分的更多信息，可以查看 [git worktrees](https://developers.openai.com/codex/app/worktrees)。

适合自动化的任务包括：

- 总结最近的 commits
- 扫描潜在 bug
- 起草发布说明
- 检查 CI 失败
- 生成站会摘要
- 按计划运行重复性分析流程

一个实用的经验法则是：skills 负责定义“怎么做”，automations 负责定义“什么时候做”。如果某个流程仍然需要大量引导，就先把它打磨成 skill；一旦它已经足够稳定可预测，自动化就能把它的价值进一步放大。

> 自动化不只是为了执行任务，也适合用来做回顾和维护。你可以定期回顾最近的会话，汇总反复出现的摩擦点，并持续改进提示词、指令或工作流设置。

## 用会话控制管理长时工作

Codex 会话不只是聊天记录。它们本质上是会随着时间不断积累上下文、决策和行动的线程，因此，会话管理本身就会直接影响结果质量。

在 Codex 应用里，会话管理最直观，因为你可以置顶线程，也可以创建 worktrees。如果你使用 CLI，下面这些 [slash 命令](https://developers.openai.com/codex/cli/slash-commands) 会很有用：

- `/experimental`：切换实验特性，并写入你的 `config.toml`
- `/resume`：恢复一个已保存的会话
- `/fork`：在保留原始 transcript 的同时创建新线程
- `/compact`：当线程变长时，生成早期上下文的摘要版本。注意，Codex 也会自动帮你压缩会话
- `/agent`：当你在并行运行多个代理，并且想在活跃代理线程之间切换时使用
- `/theme`：选择语法高亮主题
- `/apps`：直接在 Codex 中使用 ChatGPT 应用
- `/status`：检查当前会话状态

尽量让一个线程只承载一个连贯的工作单元。如果手头的工作仍然属于同一个问题，就通常应该留在同一个线程里，因为这样可以保留推理链路。只有当工作真正分叉时，再去 fork 新线程。

你也可以使用 Codex 的 [subagent](https://developers.openai.com/codex/concepts/subagents) 工作流，把边界清晰、范围明确的任务从主线程拆出去。这样主代理可以专注于核心问题，而 subagent 则负责探索、测试或分类等辅助任务。

## 常见错误

以下是第一次使用 Codex 时常见、也值得尽量避免的几个错误：

- 在 `AGENTS.md` 或 skill 里塞入过多持久规则，而不是把任务相关内容留在提示词中
- 不让 Codex 看到它要完成的工作环境，比如不告诉它怎样才是运行构建和测试命令的正确方式
- 在复杂、多步骤任务中跳过规划
- 在还没理解工作流之前，就直接给计算机完全权限
- 在同一个文件上让活跃线程并行工作，却不使用 git worktrees
- 在流程还不可靠时，就急着把重复任务自动化
- 把 Codex 当成必须全程盯着看的工具，而不是一个可以在你工作时并行运转的助手
- 以“每个项目一个线程”而不是“每个任务一个线程”的方式组织会话，导致上下文不断膨胀、结果逐渐变差
